  Setting up your environment: BitBucket and SourceTree

This module guides you through setting your environment for source code management in the course. Please try and do this at home before coming to the lab. If you have problems the tutor will be able to help.

You will need to use several windows during the set up process: one browser window for the instructions you are reading now, one browser window for bitbucket, one window for sourcetree and one for a text editor or commands. Make all windows wide enough to see instructions and get the full view so they match the instructions below.

The first step in the process of setting up your environment is to create a BitBucket account. This is very easy: just go to https://bitbucket.org and "Sign up for free".
NOTE: please use your Unikey email address (eg WXYZ4321@uni.sydney.edu.au) when signing up for a free account.
This will cause your account to be converted to an unlimited account that can be used for groups larger than 5.

Step two is to install the Atlassian "SourceTree" application. Go to http://sourcetreeapp.com/ and download and install the right version for your operating system (windows or Mac). If you use SourceTree on the SIT lab computers you will find that it is already installed. However, you need to go to the "Tools" menu, select "Mercurial" and then click the "enable" button.
Create a bitbucket hg repository on the local machine and at the central repository

After you have created your account and installed SourceTree, login in to bitbucket and press the "create" button in the top menu bar to create a new repository.

The following screen will appear:

Fill in the name and description and be careful to tick the boxes as shown. Then click on the "Create repository" button at the bottom.

After creating the repository the next step is to set it up.

We're going to use the "SourceTree" repository client app running on your local machine so just click the "Clone in SourceTree" button.
The SourceTree program should start and give you a window like this:

Click on "Clone" and SourceTree will make a copy of your repository on your local machine. SourceTree allows you specify the destination directory or path to store the repository files locally.

Now it's time to create a file in your new repository. Use any text editor you prefer, create a file called "README" in the directory you made for the respository. The convention is that a README file contains information about the repository.

The file will appear in a SourceTree window like this:

Notice that the file README.txt has a question mark next the name. This indicates that the file is in the repository directory but not yet incorporated into the repository database.
To add the file to the repository database just tick the box next to the file and click on the button on the top menu bar.
Now you have some pending changes, namely that you have a file to add to the repository. To commit them:
  1. Click the SourceTree commit button on the top row
  2. Complete the requested information about your name and email
  3. Fill in the comment box and click the commit button below it.
You should always add a comment that explains briefly what changes you've made to your repository. Later you can quickly find details of changes made.
The changes have now been recorded in your local repository. If you wanted you could stop at this point and just work with your text and the local repository. But that means your text is locked up on the local machine. If you work with a group they can't get access. If the repository is on the University system, you don't have access on your laptop.

The solution to these problems is to use the BitBucket repository to keep a copy of all your changes. You do this by "push"ing the set of changes to BitBucket.

Notice there is a button labeled "push" with a red 1 overlaid? Clicking that button will "push" the changes to the BitBucket server. The 1 means there is one change set that hasn't been sent to the server.

A screen like that at the right should appear at this point. Click on OK to complete the push.

Let's leave the SourceTree program for a moment and see what has happend on the BitBucket server. Go to the bitbucket site and login. Then click on the "Source" button on the left. You should see the README file that was created earlier, saved in the repository and then pushed to the server.

The next thing to demonstrate is how to change a file and save the change in the repository.
Using your text editor, change the README file in some way. Iv'e just added an extra line. Make sure you save the changed file.
The SourceTree program has noticed that a file has changed and will display what the changes are.

The next step is to commit the change by pressing the "commit" button on the top left.

SourceTree will now give you the opportunity to add a detailed description of the change. Finally, press the "commit" button on the lower right to commit the change.
Now we can push the change to bitbucket as before.
Go back to the bitbucket web page and click "Overview" on the left. This will show the README file and list the commit's that have been made to the repository (on the right).
If you click on one of the commits, for example the most recent, you will see the details of what was changed.

What is Bitbucket?

Since all the report marks and all the individual marks require evidence that is in your Bitbucket Wiki, it is critical that each group and each individual learns to use the system effectively.

Bitbucket provides several tools, each with a different role:

  • mercurial - version control and distribution of code between members.
  • Wiki - sharing of documentation and resources.
  • Issue Tracker - assignment of tasks to contribute towards project milestones.

How do I access my bitbucket repository?

After you have created your bitbucket account (for example, with username USERNAME), you can access all your repositories with the URL: https://bitbucket.org/USERNAME

Is there a Mercurial client as an alternative to the command line?
  • SourceTree is an app for Windows or Mac that will help you manage your repository.
  • TortoiseHG is a similar application, also available for Windows and Mac but also Linux.

Issue Tracking - how big as an issue ticket?

Tickets are for small self-contained tasks relating to the goals of your project. You should create and assign each ticket to single group member, and that person should use them to record details of their progress. Tickets should be closed when the task is completed or resolved.
